# Architecture Decision Criteria

Defines the selection criteria for sync vs async, API design, data access patterns, dependency injection, and project structure.

---

## 1. Sync vs Async Decisions

### When to Use Async

| Criteria | Sync | Async |
|---------|------|-------|
| CPU-bound computation | Default | Not beneficial (use multiprocessing instead) |
| I/O-bound with few concurrent requests | Acceptable | Optional |
| I/O-bound with many concurrent requests | Bottleneck | Recommended |
| WebSocket / streaming | Not practical | Required |
| Database queries (typical web app) | Acceptable with thread pool | Preferred with async drivers |
| External API calls | Acceptable for simple cases | Recommended for multiple calls |

**Principle**: Default to async for web applications (FastAPI is async-first). Use sync for CLI tools and scripts unless concurrency is needed. Never mix sync and async database drivers in the same project.

### Event Loop Considerations

| Decision Point | Choice | Reason |
|---------------|--------|--------|
| Async framework | asyncio (stdlib) | Standard, well-supported |
| Running CPU-bound work in async context | `run_in_executor` with ProcessPoolExecutor | Avoid blocking the event loop |
| Async testing | pytest-asyncio | Consistent with pytest ecosystem |

## 2. API Design Decisions

### REST vs GraphQL vs gRPC

| Use Case | REST | GraphQL | gRPC |
|----------|------|---------|------|
| Public API for external consumers | Recommended | Acceptable | Not recommended |
| Internal microservice communication | Acceptable | Not recommended | Recommended |
| Mobile clients with bandwidth concerns | Acceptable | Recommended | Acceptable |
| Simple CRUD applications | Recommended | Overkill | Overkill |
| Complex data relationships with variable queries | Acceptable | Recommended | Not recommended |

**Default**: REST with OpenAPI (auto-generated by FastAPI). Only introduce GraphQL or gRPC when there is a concrete need.

### URL and Endpoint Design

| Decision Point | Choice | Reason |
|---------------|--------|--------|
| URL style | `/api/v1/resources/{id}` | RESTful, versioned |
| Naming convention | Plural nouns for collections | Consistency (`/users`, not `/user`) |
| Nested resources | Max 2 levels deep | `/users/{id}/orders`, not deeper |
| Filtering/sorting | Query parameters | `/users?status=active&sort=-created_at` |
| Bulk operations | Dedicated endpoints | `POST /users/bulk` instead of multiple single calls |

## 3. Data Access Pattern Decisions

### Repository Pattern vs Direct ORM

| Criteria | Repository Pattern | Direct ORM in Service |
|---------|-------------------|----------------------|
| Project size > 5 models | Recommended | |
| Need to swap databases | Required | |
| Testing with mocks/fakes | Recommended | Possible but messy |
| Simple CRUD, < 5 models | | Acceptable |
| Rapid prototyping / MVP | | Acceptable |

**Principle**: Use the Repository pattern by default. It provides a clean boundary between business logic and data access, making testing straightforward.

### SQLAlchemy Session Management

| Context | Strategy | Implementation |
|---------|----------|----------------|
| FastAPI request | Session-per-request | Dependency injection via `Depends(get_db)` |
| Background tasks | Explicit session creation | `async with AsyncSession() as session:` |
| CLI commands | Scoped session | Context manager wrapping the command |
| Tests | Transaction rollback | Create session, begin transaction, rollback after test |

## 4. Dependency Injection Decisions

### DI Approach

| Criteria | FastAPI `Depends` | Manual Constructor Injection | DI Container (e.g., dependency-injector) |
|---------|------------------|-----------------------------|-----------------------------------------|
| Route handler dependencies | Recommended | | |
| Service-to-service dependencies | | Recommended | Acceptable for large projects |
| Configuration injection | | Recommended | Acceptable |
| Test overrides | Easy (dependency_overrides) | Easy (pass mocks) | Easy (override providers) |

**Principle**: Use FastAPI's built-in `Depends` for route-level injection. Use constructor injection for service-layer dependencies. Only introduce a DI container library when the project has 20+ injectable services.

## 5. Project Structure Decisions

### Directory Layout

| Structure Style | When to Use | Example |
|----------------|-------------|---------|
| Flat (by layer) | Small projects, < 5 domains | `routes/`, `services/`, `models/`, `schemas/` |
| Domain-based (by feature) | Medium-large projects, 5+ domains | `features/users/`, `features/orders/`, each with own routes, services, models |
| Hybrid | Large projects with shared infrastructure | `core/` for shared, `features/` for domains |

### Module Placement

| Module Type | Location | Modifiable? |
|-------------|----------|-------------|
| Pydantic schemas (API contracts) | `schemas/` or `features/*/schemas.py` | Versioned changes only |
| SQLAlchemy models | `models/` or `features/*/models.py` | Via Alembic migrations |
| Business logic | `services/` or `features/*/service.py` | Freely within domain |
| Route handlers | `routes/` or `features/*/routes.py` | Thin wrappers only |
| Shared utilities | `core/` or `lib/` | Requires review |

### File Splitting Decisions

| Condition | Decision |
|------|------|
| Module exceeds 300 lines | Consider splitting. 300 lines is the point where a module typically stops being easy to reason about as a unit. It's a review trigger, not a hard rule — a 350-line module with clear linear flow is fine; a 200-line module with 5 classes and complex interactions should be split. |
| Logic that needs to be independently testable | Separate into its own module |
| Same Pydantic model used in multiple features | Move to `schemas/shared.py` or `core/schemas.py` |

## 6. Validation Placement Decisions

### Pydantic and SQLAlchemy Responsibility Split

| Case | Responsibility | Reason |
|--------|----------|------|
| API input validation rules | Pydantic only | Changes frequently |
| DB integrity constraints (PK, FK, UNIQUE, NOT NULL) | SQLAlchemy + DB only | Cannot be fully expressed in Pydantic |
| Shared business rules (allowed values, min/max) | Both | Type-safe double defense |
| Data transformation (None -> default value, date parsing) | Pydantic only | API-specific transformation |

## 7. Testing Strategy Decisions

### Test Target Priority

```
1. Authentication/authorization (sessions, IDOR)     <- Highest: a flaw here means full data breach
2. Payment processing                                <- Financial loss is immediate and concrete
3. DB operations + API endpoints                     <- Data corruption is hard to reverse
4. Business logic (including Pydantic schemas)        <- Domain bugs erode user trust
5. User-critical API flows                            <- Broken flows = lost revenue
6. External API integrations                          <- Third-party changes break silently
7. Edge cases                                         <- Long-tail reliability
```

### Test Type Selection

| Test Type | Purpose | Tool |
|----------|------|--------|
| Unit | Pure functions, business logic | pytest |
| Integration | Service layer, API endpoints, SQLAlchemy | pytest + httpx (TestClient) |
| E2E | Full user journeys (API sequences) | pytest + httpx or Playwright (if UI) |
| Contract | API contracts with external services | Pact / schemathesis |
| Performance | API load, response time | k6 / locust |
| Security | Authentication, authorization, injection | ZAP, bandit, safety |

### Test Database Strategy

- Use a test-dedicated SQLite (file or in-memory) or PostgreSQL test database
- Apply the same schema as production via Alembic migrations
- Incorporate migration dry-run into CI/CD
